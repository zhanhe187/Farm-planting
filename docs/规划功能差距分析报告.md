# FPMS 当前功能与最初规划差距分析报告

> 分析日期：2026-05-15  
> 对比基准：`docs/农场种植管理系统-规划文档.md`  
> 当前源码范围：`src/main/java`、`src/main/resources/templates`、`src/main/resources/db`、`src/test/java`  
> 说明：本报告以当前代码可见能力为准，不依赖本地数据库实连调结果。用户已要求暂不改数据库结构，因此差距项中凡涉及新增表、改字段、迁移脚本的内容均列为后续数据库联调或生产化阶段处理。

---

## 一、总体结论

当前项目已经从“可演示 MVP”推进到“核心业务闭环 + AI/手机扫码真实调用链路”的阶段。最初规划中的 C1、C3、C4、C5、C7、C10、C11、C12 均已有可运行代码或主要链路；其中 AI 对话、AI 端点配置、手机扫码拍照识别已经接入 OpenAI 兼容协议，不再是纯展示页面。

但距离最初规划中的完整生产级系统仍有明显差距，主要集中在：

- 权限体系仍是 Session 登录 + 角色字段展示，未完成 Sa-Token、菜单权限、按钮权限和行级数据权限。
- 持久层仍以 JdbcTemplate 为主，未按规划迁移到 MyBatis-Plus Mapper、分页插件、数据权限插件、乐观锁插件。
- 销售、订单、回款、客户后台、报表导出等经营闭环功能基本未实现。
- 操作日志、农事日历、定时任务、库存预警通知、公开页限流、AI 限流、AI 主备熔断等生产化能力尚未完成。
- AI 目前支持真实调用，但多轮会话、历史摘要、流式逐 token、图片压缩、结果缓存、一键加入作物库等规划细节仍未落地。
- 手机端目前覆盖“扫码绑定 + 拍照识作物”，未覆盖完整 PWA、拍照记作业、拍照记病虫害、待办任务、GPS 地块联动。

## 四、技术栈与架构差距

### 4.1 已符合规划的技术点

- Spring Boot 2.7、Thymeleaf、Maven、SQL Server Driver 已使用。
- JdbcTemplate 已形成清晰服务层，适合当前 MVP 快速落地。
- zxing 依赖已用于手机绑定二维码生成。
- WebFlux 依赖存在，但当前 AI 调用实际使用 `RestTemplate`。
- Jackson 已用于 AI Vision JSON 解析。
- AES-GCM 已用于 AI API Key 加密，未配置主密钥时保留本地开发可用模式。
- JUnit 5 + Mockito 测试覆盖状态机、安全间隔期、库存扣减、Prompt 脱敏、AI Provider、AI 客户端、手机上传识别等关键点。

### 4.2 与规划不一致或未完成的技术点

| 技术规划 | 当前状态 | 差距 |
| --- | --- | --- |
| MyBatis-Plus 全面持久层 | 依赖存在，但业务使用 JdbcTemplate | 缺 Mapper、Entity、分页、乐观锁、数据权限插件落地。 |
| Sa-Token 鉴权 | 未使用 Sa-Token，采用 Session 拦截 | 缺注解鉴权、权限码、移动端专用会话、Token 风格登录。 |
| LayUI | 当前为自定义 Thymeleaf + CSS | 未引入 LayUI 组件体系。 |
| ECharts 多图表 | 仅基础看板/状态展示 | 缺规划中的多维 BI 图表。 |
| WebClient + Reactor 流式 AI | 当前 `RestTemplate` 同步调用 | 缺对模型 SSE 的逐 token 转发和中途取消。 |
| Caffeine 缓存 | 依赖存在，未用于核心功能 | 缺 AI 端点缓存、限流、识别结果缓存、字典缓存。 |
| EasyExcel | 依赖存在，功能未实现 | 缺农资、采收、报表导入导出。 |
| Thumbnailator | 依赖存在，功能未实现 | 手机图片上传未压缩，直接 base64 传给 AI。 |
| JSON Schema Validator | 依赖存在，功能未实现 | AI Vision 仅尝试解析 JSON，未做 Schema 校验与重试。 |
| Knife4j/OpenAPI | 未实现 | 缺接口文档页面。 |
| AOP 操作日志 | AOP 依赖存在，功能未实现 | 缺业务操作日志切面和异步落库。 |

## 五、业务模块差距详单

### 5.1 登录、用户与权限

已实现：

- `AuthController` 支持用户名密码登录。
- `PasswordService` 支持密码校验。
- `SessionInterceptor` 保护后台页面，公开 `/trace/**`、`/m/**` 等路径。
- `SessionUser` 保存用户 ID、用户名、显示名、角色、data_scope。

差距：

- 未实现用户、角色、菜单、权限维护页面。
- 未实现 SUPER_ADMIN、FARM_OWNER、AGRI_TECH、WAREHOUSE、FIELD_WORKER、CUSTOMER 的功能级差异。
- 未实现数据权限过滤，同一 SQL 查询不会根据 data_scope 自动裁剪数据。
- 未实现 Sa-Token、Token 续期、移动端独立 token、权限注解。

建议优先级：高。权限是生产化前必须补齐的底座。

### 5.2 基础档案

已实现：

- SQL 脚本包含 `farm_plot`、`farm_crop`、`farm_material`、`stock_inventory`。
- 页面覆盖地块、库存、批次基础列表。

差距：

- 作物、农资、地块缺完整新增/编辑/删除/停用流程。
- 缺供应商、农资批号、成本价、有效期、库存预警处理。
- 缺导入导出和字段校验。

建议优先级：中。

### 5.3 农事与库存

已实现：

- 农事作业新增。
- 作业关联农资消耗并扣减库存。
- 生成 `stock_out_order` 出库流水。
- SQL 触发器脚本保障 `plant_operation` 不可更新/删除。

差距：

- 单次作业仅支持一个农资明细。
- 缺红冲修正入口。
- 缺农事图片/附件。
- 缺任务指派、待办、日历。
- 缺库存并发冲突重试和业务锁定。

建议优先级：高。多农资明细和红冲是业务可信度的关键。

### 5.4 采收、销售与追溯

已实现：

- 采收前安全间隔期校验。
- 采收成功后生成 trace_code。
- 公开追溯页查询批次、地块、作物、作业时间线。

差距：

- 销售、订单、客户、回款完全未成体系。
- trace_code 规则较简化，未生成二维码文件。
- 公开页未实际使用字段白名单控制脱敏。
- 缺检测报告、产地介绍、公开页限流。

建议优先级：高。销售/回款是从“种植管理”走向“经营管理”的主要缺口。

### 5.5 看板与报表

已实现：

- 驾驶舱 KPI：地块数、在种批次、待采收、低库存。
- 最近作业、批次状态、库存数据。
- 地块 Canvas 布局展示。

差距：

- 未实现产量同环比、收益分析、农资消耗结构、地块产出强度、病虫害分布。
- SQL Server 物化视图脚本存在，但未接入服务层。
- 缺筛选器、日期区间、导出、图表解释。

建议优先级：中。若用于答辩或演示，BI 视觉效果值得尽早增强。

### 5.6 AI 对话与端点配置

已实现：

- 管理员可在 `/providers` 新增 AI 端点。
- API Key 服务端保存并脱敏展示。
- 有 `FPMS_AI_MASTER_KEY` 时可 AES-GCM 加密；未配置时支持本地开发。
- `CHAT` 和 `VISION` 按场景路由到不同端点。
- `/ai/chat` 调真实 OpenAI 兼容接口。
- 连通性测试会真实请求端点并写入 `ai_call_log`。

差距：

- 端点列表缺编辑、禁用、删除、重置 Key 等管理动作。
- 同场景多个端点只按优先级取第一个，不支持失败自动切备用。
- `ai_call_log` 未记录 token 数、请求体大小、模型返回错误码细节。
- 缺用户级/全局限流。
- 缺多轮会话和历史记录。
- SSE 接口只是一次性推送完整答案，未实现真正流式 token。
- Prompt 上下文只注入全场概况和可选批次摘要，未注入最近 3 次作业、库存风险、天气、安全间隔期等更丰富业务信息。

建议优先级：高。真实 AI 已接上，下一步应补“稳定性、成本控制、上下文质量”。

### 5.7 手机端扫码与拍照识别

已实现：

- `/mobile` 生成一次性 token 和二维码。
- `/m/bind` 校验 token，绑定移动端 Session。
- `/m/recognize` 接收手机拍照/选图上传。
- 图片校验大小和 MIME，限制 JPG/PNG/WebP。
- 调用 `VISION` 场景 AI 端点，解析作物名称、用途、培育方法等结果。

差距：

- bind_token 存在内存中，服务重启即丢失；未使用 `ai_bind_token` 表。
- token 使用状态不是数据库原子置位，多实例部署不可用。
- 图片未压缩，可能增加调用成本。
- 未写 `ai_recognize_log`，没有图片 hash 缓存。
- 未实现一键加入 `farm_crop`。
- 未实现拍照记作业、拍照记病虫害、我的待办、GPS 定位地块。
- 移动端只是轻量 H5，不是完整 PWA。

建议优先级：中高。当前链路已能演示，生产化重点是持久化、安全和降本。

## 七、建议后续路线图

### P0：不改数据库结构前可继续做

1. 完善 AI 错误提示和降级文案，避免接口异常直接暴露技术栈。

5. 手机端结果页增加“确认识别结果”按钮，但暂不写库。
6. 增加 MockMvc 测试覆盖关键页面流程。

### P2：需要生产化改造或可能涉及结构扩展

1. 引入 Sa-Token 并实现完整 RBAC + 行级数据权限。

如果目标是“课程设计/项目验收”，建议优先补：

1. AI 对话流式输出。
2. 手机拍照识别结果写入日志和一键加入作物库。
3. 地块布局图悬浮详情和状态着色。
4. 追溯公开页二维码与字段脱敏。
5. 简单销售记录和收益看板。

如果目标是“真实农场试用”，建议优先补：

1. 权限体系和数据权限。
2. 基础档案完整 CRUD。
3. 农事多农资明细和红冲修正。
4. bind_token/识别日志持久化。
5. SQL Server 实库联调、备份和部署脚本。

## 九、结论

当前版本已经具备核心演示价值：登录后台、看地块和批次、录农事、扣库存、采收校验、公开追溯、AI 对话、AI 端点配置、手机扫码拍照识别。它已经覆盖最初规划中最有辨识度的主线。

但它还不是完整生产版。后续最大的差距不是“有没有页面”，而是“权限、数据持久化一致性、经营闭环、AI 稳定性和移动端完整工作流”。建议下一阶段按 P0/P1/P2 分层推进，先在不改数据库结构的前提下把已有表和已有入口用扎实，再进入权限和销售等较大模块。
