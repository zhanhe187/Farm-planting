# 智慧农场种植管理系统 —— 项目规划文档

> 项目代号：**FPMS**（Farm Planting Management System）  
> 编写日期：2026-05-14  
> 数据库：**Microsoft SQL Server 2019+**  
> 架构基线：Spring Boot 2.7 + MyBatis-Plus + Sa-Token + Thymeleaf + LayUI + ECharts  
> 设计立场：面向中小型种植主体，以**完整农事生命周期 + 食品安全合规 + AI 智能化**为核心竞争力，覆盖 PC 与移动端。

---

## 一、项目概述

### 1.1 项目背景
中国家庭农场和合作社数量已超 600 万家，但管理工具仍以纸质 + Excel 为主，痛点集中在：
- **种植档案缺失**：哪块地什么时候种了什么，全靠记忆
- **食品安全合规压力**：监管要求"从田头到餐桌"全程可追溯，传统方式做不到
- **农资浪费**：化肥、农药用量凭经验，库存账实不符
- **决策无数据**：不知道哪种作物最赚钱、哪块地最贫瘠
- **多农场难协同**：连锁化经营时数据无法横向对比

### 1.2 项目愿景
做一套**面向中小型种植主体**的 Web 管理系统，让一个不懂 IT 的农场主也能：
- 一张图看清所有地块当前在种什么
- 一个时间线追踪每个批次的全生命周期
- 一张二维码给消费者展示完整溯源
- 一个看板掌握产量、农资、收益的关键数据
- 一套预警机制避免食品安全事故

### 1.3 用户角色
| 角色 | 角色码 | 主要职责 |
|------|--------|----------|
| 超级管理员 | SUPER_ADMIN | 系统配置、租户/用户/权限维护 |
| 农场主 | FARM_OWNER | 决策审批、看板查看、报表导出 |
| 农技员 | AGRI_TECH | 种植计划、农事指导、品质把控、病虫害诊断 |
| 仓库管理员 | WAREHOUSE | 农资出入库、盘点、库存预警处置 |
| 田间工人 | FIELD_WORKER | 录入作业、上传图片、领取任务 |
| 客户/经销商 | CUSTOMER | 查看可售批次、下单、查回款（精简后台） |
| 公众访客 | （免登录） | 通过追溯码查看溯源信息 |

### 1.4 项目边界
**本期 MUST**：地块/品种/农资基础档案、种植批次全流程、农事记录、库存与单据、采收与销售、追溯码、数据看板、RBAC + 数据权限、**AI 对话助手、AI 端点动态配置、手机扫码识别作物**。  
**本期 SHOULD**：农事日历、操作日志、定时预警、地块布局图、AI 病虫害二次分析。  
**本期可选**：移动端工人录入完整 PWA、IoT 传感器接入、多租户 SaaS 化、AI 历史会话云同步。

## 二、核心特色与亮点（差异化设计）

本项目围绕「农事生命周期 + 食品安全合规 + 数据驱动 + AI 智能化」形成 **12 项核心特色**，构成系统的差异化竞争力。

### 2.1 特色速览
| 编号 | 特色 | 价值定位 |
|------|------|----------|
| C1 | **批次生命周期状态机** | 把每次种植抽象为有完整生命周期（计划→播种→生长→采收→完结/废弃）的动态对象，杜绝"一笔糊涂账" |
| C2 | **作业事件流（Event Sourcing 简化版）** | 每个批次累积一条**不可篡改的农事事件流**，构成溯源可信度的根基 |
| C3 | **农资双联动**（库存↔作业） | 施肥/打药等作业**自动消耗**农资库存，业务闭环、账实相符 |
| C4 | **农药安全间隔期校验** | 采收时**强制**校验最近一次施药+间隔期，未达标禁采，体现行业纵深 |
| C5 | **追溯码 + 免登录公开页** | 通过白名单字段控制脱敏，让消费者一码看清产品来源 |
| C6 | **数据权限（行级权限）** | 按"地块负责人 / 作业指派人"实现行级数据隔离，多人协作不串号 |
| C7 | **地块平面布局图（Canvas 可视化）** | 可拖拽地块布局 + 产量热力叠加，决策一图全览 |
| C8 | **轻量 BI 看板**（多维分析） | 同比/环比、品种对比、地块产出强度等多维度图表 |
| C9 | **物化视图缓存核心报表** | 利用 SQL Server **Indexed View** 物化关键报表查询，应对历史数据增长 |
| **C10** | **AI 农事问答助手（流式对话）** | 内置 AI 对话区，支持业务上下文注入与流式输出，给出可操作的农事建议 |
| **C11** | **AI 端点动态配置（Provider/URL/Key 可热切）** | 管理员后台维护多套 AI 端点，业务侧按场景路由，支持主备熔断 |
| **C12** | **手机扫码连接 + 拍照 AI 识作物** | 临时绑定码把手机变成专用录入终端，拍照即得作物百科与培育建议 |

### 2.2 重点特色展开

#### C1：批次生命周期状态机
- 状态：`PLANNED → SOWED → GROWING → READY_HARVEST → HARVESTING → COMPLETED`，分支 `ABANDONED`
- 每次状态推进都校验前置条件（如未播种不允许采收），并写入 `batch_status_log` 留痕
- 用 `enum + Map<from, allowedTo>` 显式建模，单测覆盖所有合法/非法迁移

#### C2：作业事件流（Append-only）
- `plant_operation` 表设计为**只追加、不更新**（业务上的不可变性）
- 表结构上加 `is_locked` 标记，状态进入 `READY_HARVEST` 后历史作业全部锁定
- 误录用"红冲单"补偿（再插入一条 `type=CORRECTION` 的反向作业），保证历史档案不被篡改
- 这是溯源可信的根基

#### C3：农资双联动
- 录入农事作业时，前端勾选消耗的农资明细 → 后端在**同一事务**内：
  1. 写 `plant_operation` 主表 + `plant_operation_material` 明细
  2. 调用 `MaterialService.consume()` 扣减 `stock_inventory`
  3. 写 `stock_out_order`（type=领用，关联 operation_id）
- 任一步失败整体回滚，杜绝"扣了库存没记作业"或反之

#### C4：安全间隔期校验
- `farm_material` 表存 `safe_interval_days`（农药打药与采收的最少间隔天数）
- 采收登记时计算：
  ```
  禁采截止日 = MAX(最近一次该批次施药日 + 该农药 safe_interval_days)
  if 当前日期 < 禁采截止日 → 抛 BusinessException 并写入合规告警
  ```
- 同时提供 **采收预检接口** `/harvest/precheck`，前端在采收页加载时预先提示

#### C5：追溯码 + 公开页
- 采收完成时生成 32 位 UUID 追溯码，并预渲染二维码 PNG（zxing）
- 公开页 `/trace/{code}`：
  - 头部：作物图片、批次号、产地、采收日期
  - 时间线：脱敏后的农事记录（保留类型/日期/工人昵称，**不展示具体农药品牌、剂量、成本**）
  - 农场介绍 + 检测报告（如有）
- 关键安全点：**绝不复用业务实体**，单独建 `TracePublicVO`，通过 `public_fields` 配置每个字段是否对外公开

#### C6：行级数据权限
- 角色：超管/老板可见全部；农技员只看自己负责的地块；工人只看自己被指派的作业
- 实现思路：
  - `sys_user` 增加 `data_scope` 字段（ALL / OWN_PLOT / ASSIGNED_TASK）
  - 通过 MyBatis-Plus 的 **数据权限插件**（`DataPermissionInterceptor`）在 SQL 末尾自动追加 `WHERE owner_id = ?`
  - 不能用 ORM 自动拼的，统一在 Mapper 上加 `@DataScope` 自定义注解 + AOP

#### C7：地块平面布局图
- 在 `farm_plot` 中存 `layout_x / layout_y / width / height`（农场坐标系，单位米）
- 前端用 HTML5 Canvas 或 ECharts `graphic` 组件绘制：
  - 拖拽调整地块位置/大小
  - 颜色映射：当前是否在种、批次状态、产量强度
  - 鼠标悬浮弹出该地块当前批次详情卡片
- 这是答辩展示的视觉爆点

#### C8：BI 看板维度
| 图表 | 维度 |
|------|------|
| 产量同环比 | 月/季/年 × 品种 × 地块 |
| 收益分析 | 客户 × 品种 × 时间 |
| 农资消耗结构 | 化肥/农药/种子 × 月份 |
| 地块产出强度 | 单位面积产量 × 地块（结合 C7 的布局图） |
| 病虫害发生分布 | 病种 × 月份 × 地块 |

#### C9：SQL Server 物化视图
- 关键报表如"批次盈亏""年度品种产量"涉及多表 JOIN，行数增长后慢
- 使用 SQL Server 特性：`CREATE VIEW ... WITH SCHEMABINDING` + `CREATE UNIQUE CLUSTERED INDEX` 物化视图
- 应用层通过专用 Mapper 直接查物化视图，毫秒级响应

#### C10：AI 农事问答助手（流式对话）
- **入口**：管理后台右下角悬浮 AI 按钮，点开为侧滑抽屉聊天面板；同时支持独立全屏页 `/ai/chat`
- **能力**：
  - 通用农事问答：施肥配比、病虫害识别建议、采收时机判断、市场行情解读
  - **业务上下文注入**：进入某批次详情后开 AI 对话，系统自动把"该批次品种、生长天数、最近 3 次作业、地块面积"作为系统提示词附带过去，让 AI 给出针对性回复
  - 多轮会话、历史保留（`ai_chat_session` + `ai_chat_message` 双表）
  - 支持 SSE / WebSocket **流式输出**，逐字渲染，体感接近原生 AI 应用
  - **快捷指令模板**：「分析这个批次的盈亏」「给我一份本周农事建议」「生成对外宣传文案」
- **技术要点**：
  - 后端通过 `RestTemplate`/`WebClient` 转发到 C11 配置的 AI 端点，**密钥不下发前端**
  - 提示词模板化：`PromptBuilder` 组装 `system / context / user` 三段
  - 限流：每用户每分钟最多 N 次调用（Caffeine 滑动窗口），可在系统参数中调整
  - 失败降级：AI 不可用时返回友好提示，不影响主业务
  - 数据合规：会话默认仅保留摘要 30 天；用户可主动清除

#### C11：AI 端点动态配置（管理员功能）
- **配置维度**（`ai_provider` 表）：
  - 名称、Provider 类型（OpenAI 兼容 / Anthropic / 自建模型 / 阿里通义 / 文心一言）
  - **Base URL**（如 `https://api.openai.com/v1`）
  - **API Key**（数据库以 AES 加密存储，回显脱敏只露后 4 位）
  - 默认模型名（如 `gpt-4o-mini`、`qwen-turbo`）
  - 适用场景（`SCENE`：CHAT 对话 / VISION 图像识别 / EMBEDDING 向量）
  - 超时、最大 tokens、温度等可选参数
  - 启用状态、是否为该场景的默认端点
- **能力**：
  - 同一场景可配置多套端点，支持**优先级排序**与**故障切换**（主端点超时自动转备）
  - 在线**连通性测试**按钮：发一条 ping 消息验证 Base URL + Key + 模型 三件套
  - 配置变更**热生效**，无需重启（通过 `AiProviderRegistry` 单例 + Caffeine 缓存监听）
  - 调用日志（`ai_call_log`）：耗时、token 用量、错误码，便于成本核算
- **安全要点**：
  - API Key 字段独立加密表 + 应用启动从 KMS/环境变量读取主密钥
  - 后台编辑页明文不回显，仅在新建/重置时填写
  - 仅 SUPER_ADMIN 角色可见 AI 端点配置菜单
  - 所有 AI 调用强制走服务端转发，**前端永远拿不到 Key**

#### C12：手机扫码连接 + 拍照 AI 识作物
- **使用流程**：
  1. PC 后台点击「手机端」按钮 → 弹出二维码（5 分钟有效，内含一次性 `bind_token`）
  2. 手机扫码 → 浏览器跳到移动端 H5 页 `/m/bind?token=xxx` → 后端校验 token → Sa-Token 签发 H5 专用会话 → 自动绑定到当前 PC 会话
  3. 手机端首页：「拍照识作物」「拍照记作业」「拍照记病虫害」「我的待办」
  4. 「拍照识作物」→ 调用手机相机 → 选择/拍摄图片 → 上传 → 后端走 C11 中 `SCENE=VISION` 的 AI 端点
  5. AI 返回：**作物名称、品种细分、植物百科介绍、培育方法、适宜温度区间、土壤偏好、常见病虫害、参考产量**
  6. 用户确认后可一键「**加入作物库**」→ 自动写入 `farm_crop` 表（去重/合并）
- **技术要点**：
  - 二维码内容是 `https://farm.example.com/m/bind?token=xxxx`，由 zxing 实时生成
  - `ai_bind_token` 表存 token、关联 PC 会话、过期时间、是否已使用（一次性）
  - 移动端为独立 H5（轻量），不强制 PWA；技术栈 LayUI Mobile 或纯 Vanilla + Tailwind 风格 CSS
  - 图片上传走 `/m/api/recognize`：限 10MB、JPG/PNG/WebP；后端先压缩再发给 AI（节省 token）
  - **AI Vision 调用流程**：
    - 构造请求：`messages` 含 `image_url`（OpenAI 兼容协议）或 base64
    - 系统提示词约束输出为**结构化 JSON**：`{name, latin_name, intro, cultivation, suitable_temp, soil, common_pest, expected_yield_per_mu}`
    - 后端解析 JSON 失败时回退为纯文本展示
  - **结果缓存**：相同图片 hash + 同模型 24h 内复用结果，降本
  - **隐私**：图片仅短期存放，48 小时自动清理；用户可设置不上云
- **场景延伸**：
  - 拍照记作业：识别图中农事场景（施肥/打药/采收）→ 预填作业类型
  - 拍照记病虫害：识别叶片病害 → 预填 `plant_pest` 表
  - 与 C7 联动：拍照时同步记录 GPS → 自动定位到对应地块

| 密码安全 | BCrypt（成本 10） | 老用户首次登录强制改密 |

## 四、重点技术与难点深度解析

### 4.1 RBAC + 行级数据权限（双层权限）
- **菜单级**：沿用 `用户 ←→ 角色 ←→ 菜单` 三表结构，菜单表新增 `permission` 字段（如 `batch:edit`），Sa-Token 注解 `@SaCheckPermission("batch:edit")` 拦截
- **行级**：在 Mapper 方法上加 `@DataScope(plotAlias="p")`，AOP 在 SQL 末尾拼接 `AND p.owner_id = #{currentUserId}`
- 数据权限范围（data_scope）：
  - `ALL`：全部数据（超管/老板）
  - `OWN_PLOT`：仅本人为负责人的地块及其批次（农技员）
  - `ASSIGNED_TASK`：仅被指派的作业（工人）
  - `OWN_CUSTOMER`：仅本人为业务员的客户（销售）

### 4.2 批次状态机（BatchStatusMachine）
- 使用枚举 + 显式迁移表：
  ```
  PLANNED       → SOWED, ABANDONED
  SOWED         → GROWING, ABANDONED
  GROWING       → READY_HARVEST, ABANDONED
  READY_HARVEST → HARVESTING, ABANDONED
  HARVESTING    → COMPLETED
  ```
- 每次推进调用 `BatchStatusMachine.transit(batch, target)`：
  1. 校验合法迁移
  2. 校验业务前置（如 GROWING→READY_HARVEST 必须满足 `已播种天数 ≥ 品种最短生长期 × 0.8`）
  3. 写入 `batch_status_log`（含旧状态、新状态、操作人、原因）
- 单元测试：参数化测试覆盖所有合法/非法路径

### 4.3 农事事件流（Append-only 设计）
- **核心约束**：`plant_operation` 不允许 UPDATE/DELETE
- 数据库层用 SQL Server 触发器兜底：
  ```sql
  CREATE TRIGGER trg_operation_block_update ON plant_operation
  AFTER UPDATE, DELETE AS
  BEGIN RAISERROR('Operation log is immutable', 16, 1) ROLLBACK; END
  ```
- 业务层提供"红冲"机制：插入 `type=CORRECTION + ref_operation_id` 的反向作业
- 这样既保证溯源不可篡改，又允许业务上的修正

### 4.4 库存并发控制
- `stock_inventory` 表加 `version` 列（乐观锁）
- MyBatis-Plus `@Version` 注解 + `OptimisticLockerInnerInterceptor`
- 扣减失败时业务层自旋重试 3 次，仍失败抛 `BusinessException("库存扣减冲突，请重试")`
- 大宗销售出库走"先冻结、再扣减"两阶段，避免长事务

### 4.5 安全间隔期校验器（SafetyIntervalChecker）
```
boolean canHarvest(batchId, harvestDate):
    list = 查询该批次所有 type=PESTICIDE 的作业（按日期 DESC）
    forbidUntil = null
    for op in list:
        for material in op.materials where material.is_pesticide:
            interval = material.safe_interval_days
            limit = op.op_date + interval days
            forbidUntil = MAX(forbidUntil, limit)
    return harvestDate >= forbidUntil
```
- 单独成类，便于单测；典型用例：无打药、单次打药、多种农药、不同批次混淆等
- 提供两种调用：`precheck`（页面提示）和 `enforce`（提交时强校验）

### 4.6 追溯码生成与公开页
- 追溯码：`F` + 6 位农场 ID + 8 位日期 + 6 位 BASE36 自增（如 `F000017-20260514-0000A1`），可读性优于纯 UUID
- 二维码：采收完成后 zxing 生成 PNG 存到 `static/upload/trace/{yyyyMM}/{code}.png`
- 公开页流程：
  1. `/trace/{code}` → 通过 `code` 查 `trace_code` 表
  2. 关联 `plant_batch` → `farm_plot` → `farm_crop` → `plant_operation`（按 `public_fields` 白名单脱敏）
  3. 渲染 Thymeleaf 模板，无需登录
- 防爬：单 IP 每分钟限 30 次（用 Caffeine 滑动窗口）

### 4.7 物化视图（SQL Server 特性）
- 创建带索引的视图：
  ```sql
  CREATE VIEW dbo.v_batch_profit WITH SCHEMABINDING AS
  SELECT b.id AS batch_id, b.code, b.crop_id,
         SUM(...) AS revenue, SUM(...) AS material_cost, ...
  FROM dbo.plant_batch b ...
  GROUP BY b.id, b.code, b.crop_id;
  CREATE UNIQUE CLUSTERED INDEX ix_v_batch_profit ON dbo.v_batch_profit(batch_id);
  ```
- 看板/盈亏报表查询直接走视图，毫秒级
- 视图自动随基础表更新而同步（满足 SCHEMABINDING 约束）

### 4.8 操作日志（AOP + 异步）
- 自定义注解 `@OperationLog(module="批次", action="状态推进")` 标注 Controller 方法
- 切面在方法返回后异步（`@Async`）写入 `sys_log`：用户、模块、动作、入参摘要、IP、UA、耗时、是否成功
- 异步用 Spring 自带 `ThreadPoolTaskExecutor`，避免阻塞业务

### 4.9 文件上传安全
- 校验：扩展名白名单 + MIME 检测 + 文件大小上限 + 图片像素上限
- 存储：UUID 重命名，按 `yyyy/MM/dd` 分桶
- 访问：通过 `/file/view?id=xxx` 间接访问，不暴露真实路径，避免目录穿越

### 4.10 缓存策略（Caffeine）
| 缓存对象 | TTL | 失效策略 |
|----------|-----|----------|
| 数据字典 | 1h | 字典更新时手动 evict |
| 用户菜单树 | 30min | 用户角色变更时 evict |
| 作物/农资下拉 | 1h | 档案更新时 evict |
| 物化视图查询 | 5min | 自动过期 |
| AI 端点配置 | 永久（手动 evict） | 端点新增/修改/启停时 evict |
| 图像识别结果（按 hash） | 24h | 自动过期 |

### 4.11 AI 对话与流式输出（C10 深度方案）
**1）整体调用链**
```
浏览器 EventSource ──> Controller(SseEmitter) ──> AiChatService
   │                                                    │
   │                                                    ├─ PromptBuilder：注入业务上下文
   │                                                    │   （当前批次、近 7 天农事、库存预警等）
   │                                                    ├─ AiProviderRegistry.pick(scene=CHAT)
   │                                                    │
   │                                                    ▼
   │                                              WebClient(BaseUrl + ApiKey)
   │                                                    │  POST /v1/chat/completions stream=true
   │   <───── data: {choices:[{delta:{content:"..."}}]} ─┤
   │   <───── data: [DONE] ─────────────────────────────┘
   ▼
渲染 token 流 + 落库（ai_chat_message）
```

**2）SseEmitter 关键代码骨架**
```java
@GetMapping(value="/ai/chat/send", produces=MediaType.TEXT_EVENT_STREAM_VALUE)
public SseEmitter send(@RequestParam Long sessionId, @RequestParam String content) {
    SseEmitter emitter = new SseEmitter(120_000L);            // 2min 超时
    aiChatService.streamReply(sessionId, content, emitter);   // 内部异步执行
    emitter.onCompletion(() -> log.debug("sse done"));
    emitter.onTimeout(emitter::complete);
    emitter.onError(e -> emitter.completeWithError(e));
    return emitter;
}
```
- 服务端用 Reactor `Flux<String>` 接收 SSE，逐 chunk `emitter.send(SseEmitter.event().data(token))`
- 完成后整段 `assistant` 消息一次性落库 `ai_chat_message`，避免高频写库
- 异常：捕获 `WebClientResponseException`，将友好文案推送到前端再 complete

**3）业务上下文注入（PromptBuilder）**
- System Prompt 模板：「你是 FPMS 智慧农场助手，根据下面的实时农事数据回答问题，回答尽量给出可操作建议；如果数据不足请明确说明」
- 上下文白名单：当前用户角色、当前查看批次 ID、批次基础信息、近 7 天农事记录摘要、库存预警 Top5
- 长度护栏：单次上下文 ≤ 6KB，超过按时间倒序截断；prompt 注入前做敏感字段脱敏（手机号、身份证号正则替换）

**4）多轮会话**
- `ai_chat_session` 一行一会话，含 `title`（首条用户问题前 20 字自动生成）、`scene`、`user_id`
- 取最近 N=10 条 `ai_chat_message` 作为 messages 历史；超长时执行「滚动摘要」：将更早记录用一次便宜模型压成 200 字摘要写入 `summary` 字段

**5）限流与降级**
- 用户级令牌桶：Caffeine `LoadingCache<userId, Bucket>`，每分钟 10 次、每天 200 次
- 端点级熔断：连续 5 次失败 → 自动切换备用端点（C11）；全部端点失败 → 返回友好提示「AI 助手繁忙，请稍后再试」并写 `ai_call_log`

### 4.12 AI 端点动态配置与路由（C11 深度方案）
**1）`ai_provider` 表核心字段**

| 字段 | 类型 | 说明 |
|------|------|------|
| name | NVARCHAR(50) | 端点名（如「生产-Qwen」） |
| provider_type | VARCHAR(20) | OPENAI / DASHSCOPE / DEEPSEEK / OLLAMA / CUSTOM |
| base_url | NVARCHAR(200) | 形如 `https://api.xxx.com/v1` |
| api_key_cipher | VARBINARY(512) | AES-GCM 密文 |
| api_key_iv | VARBINARY(16) | GCM IV |
| default_model | NVARCHAR(80) | 如 `qwen2.5-7b-instruct` |
| scene | VARCHAR(20) | CHAT / VISION / EMBEDDING |
| priority | INT | 同 scene 下小者先用，作主备 |
| timeout_ms | INT | 默认 60000 |
| enabled | BIT | 0/1 |

**2）密钥安全**
- 主密钥 `fpms.ai.master-key`：仅写在 `application-prod.yml` 或环境变量，不入库
- AES-GCM 加密，每条 IV 独立；保存时只接收明文 → 加密落库 → 立即从内存清空
- 列表/编辑接口返回掩码 `sk-****ABCD`；只有 `SUPER_ADMIN` 角色可访问 `/admin/ai-provider/**`，并打操作日志

**3）AiProviderRegistry 热加载**
- 启动时一次性加载到内存 `Map<scene, List<AiProvider>>`
- 增删改/启停时通过事件 `AiProviderChangedEvent` 失效缓存并重建注册表
- `pick(scene)`：按 priority 升序遍历，跳过被熔断的端点；返回不可变快照对象

**4）连通性测试**
- `POST /admin/ai-provider/test/{id}`：内部组装最简 prompt（"ping"）调用，返回 RT、HTTP 码、模型名、token 数
- 前端弹层显示绿灯/红灯，避免管理员盲配

### 4.13 手机扫码绑定 + AI 拍照识别（C12 深度方案）
**1）扫码绑定流程（一次性令牌方案）**
```
PC 端：用户点击「扫码连接手机」
       │
       ▼
   生成 bind_token (32位随机) → 存 ai_bind_token (user_id, expire_at=now+5min, used=0)
       │
       ▼
   前端 Canvas 渲染二维码：https://farm.example.com/m/bind?token=xxxx
       │
   ┌───┴────────────────────────────────────┐
   │                                        │
手机扫码                                  PC 端 5s 轮询 /ai/bind/status?token=xxxx
   │                                        │
   ▼                                        ▼
H5 打开 /m/bind?token=xxxx              已绑定 → 关闭弹层，显示「手机已连接」
   │ Sa-Token 校验 token 有效且未使用
   │ 标记 used=1，签发 H5 会话 (cookie / sessionStorage)
   ▼
跳转 /m/home → 拍照 / 我的作业 / 简报
```
- token 一次性、5 分钟过期；过期或被使用后自动作废，避免二维码截图被复用
- H5 会话与 PC 会话独立，互不影响登出
- 安全限制：仅在 HTTPS 下展示，QR 内不携带任何用户敏感信息

**2）移动端 H5 页面（LayUI Mobile + 原生拍照）**
- 入口页：作物识别 / 农事打卡 / 我的批次
- 拍照：`<input type="file" accept="image/*" capture="environment">`，触发系统相机
- 上传前压缩：移动端 JS 用 Canvas 缩放至最长边 1280px、JPG 质量 0.8（≤ 200KB）
- 服务端再次用 Thumbnailator 二次压缩与统一格式

**3）AI Vision 识别**
```
/m/api/recognize (POST multipart)
  │
  ├─ 计算 SHA-256 → 命中缓存（ai_recognize_log.image_hash）→ 直接返回
  │
  ├─ 未命中：AiProviderRegistry.pick(VISION)
  │       │
  │       ▼
  │   构造请求：messages=[{role:user, content:[{type:image_url,image_url:{url:base64...}},
  │                                            {type:text, text: SYSTEM_VISION_PROMPT}]}]
  │       │
  │       ▼
  │   接收 JSON 字符串 → JSON Schema 校验 →
  │   {name, latin_name, intro, cultivation, suitable_temp, soil, common_pest, expected_yield_per_mu, confidence}
  │
  ├─ 写入 ai_recognize_log（含原图、缩略图、hash、结果、耗时、provider_id）
  │
  └─ 返回前端：渲染卡片 + 「加入作物档案」按钮 → 触发 farm_crop INSERT
```
- **强约束 JSON 输出**：System Prompt 明确「只返回 JSON，不要任何解释；字段缺失填 null」；用 `response_format={type:"json_object"}` 强制
- **置信度阈值**：`confidence < 0.6` 时前端提示「识别可能不准，请确认」
- **失败兜底**：JSON 解析失败 → 重试一次（temperature=0）→ 仍失败则返回原始文本与友好提示

**4）成本与频控**
- 同 hash 24h 内复用结果，节省视觉模型调用
- 每用户每日上限 30 次（可配）
- 大图二次压缩到 JPG 质量 0.7、最长边 1024px 后转 base64 调用

```sql
-- 批次盈亏物化视图（示例）
CREATE VIEW dbo.v_batch_profit
WITH SCHEMABINDING AS
SELECT
  b.id AS batch_id,
  b.code,
  b.crop_id,
  b.plot_id,
  ISNULL(SUM(so_si.revenue), 0) AS revenue,
  ISNULL(SUM(mat.cost), 0) AS material_cost,
  COUNT_BIG(*) AS cnt
FROM dbo.plant_batch b
LEFT JOIN dbo.sale_order so ON so.batch_id = b.id
LEFT JOIN dbo.sale_payment sp ON sp.sale_id = so.id
-- 此处简化，真实写法需展开 sale_item 等
GROUP BY b.id, b.code, b.crop_id, b.plot_id;

CREATE UNIQUE CLUSTERED INDEX ix_v_batch_profit
  ON dbo.v_batch_profit(batch_id);
```
> 说明：SQL Server Indexed View 对 SELECT 中的聚合、JOIN、字段类型有较多约束，最终视图 SQL 在编码阶段细化，必须满足 SCHEMABINDING 与确定性。

### 6.4 SQL Server 设计要点
1. **分页**：MyBatis-Plus 分页插件方言指定 `DbType.SQL_SERVER`，底层使用 `OFFSET ... FETCH NEXT`
2. **NVARCHAR**：所有可能含中文的列统一 `NVARCHAR`，避免乱码
3. **DATETIME2**：精度高于 `DATETIME`，存储事件时间用 DATETIME2
4. **IDENTITY**：自增主键，便于 MyBatis-Plus 自动回填
5. **触发器**：用于强约束（如农事事件不可修改）
6. **物化视图**：报表加速（C9）
7. **WITH (NOLOCK)**：报表查询可视情况加，提升并发，但要接受脏读
8. **CTE + 窗口函数**：批次时间线、连续库存等查询用 CTE 简化

## 十、风险与应对

| 风险 | 影响 | 概率 | 应对策略 |
|------|------|------|----------|
| 业务建模复杂，超出工作量 | 进度失控 | 高 | 严格分级（MUST/SHOULD/可选）；课设走 5 周压缩方案 |
| 状态机分支多，迁移错误 | 数据混乱 | 中 | `BatchStatusMachine` 独立成类 + 参数化单测覆盖所有路径 |
| 库存并发扣减超卖 | 数据错误 | 中 | 乐观锁 + 自旋重试 + 失败抛业务异常；销售大单走两阶段冻结 |
| 触发器导致性能下降 | 写入慢 | 低 | 触发器只做防御性检查，不做业务逻辑；写入主键一定带索引 |
| SQL Server 物化视图约束多 | 视图建不出来 | 中 | 视图必须 `WITH SCHEMABINDING` + 确定性函数；提前用真实数据验证 |
| 安全间隔期判定错误 | 食品安全风险 | 中 | `SafetyIntervalChecker` 独立成类；典型用例覆盖；记录每次校验结果 |
| 公开溯源页暴露敏感字段 | 隐私泄漏 | 高 | 专用 `TracePublicVO` + 字段白名单；绝不复用业务实体；review 必查 |
| 公开页被恶意爬虫扫描 | 服务受影响 | 中 | Caffeine 滑动窗口限流（单 IP/min）+ 二维码加签防伪造 |
| 文件上传被滥用 | 安全漏洞 | 中 | 扩展名白名单 + MIME 校验 + UUID 重命名 + 不暴露真实路径 |
| 数据权限漏拦截 | 越权读 | 高 | `@DataScope` 切面统一处理；高敏接口加单测；review 必查 |
| 操作日志阻塞业务 | 体验差 | 中 | 异步线程池 + 队列容量保护，丢失记日志而非阻塞主流程 |
| Sa-Token 漏放公开路径 | 401 跳转登录 | 中 | `SaTokenConfig` 显式枚举；新增公开路径必须 PR 标注 |
| 密码仍明文 | 安全审计不过 | 高 | 强制 BCrypt；老用户首次登录强制改密码 |
| Canvas 布局图复杂 | 工期紧 | 中 | 用 Konva.js 简化拖拽；只支持矩形地块；MVP 阶段先做静态展示 |
| 物化视图刷新延迟 | 看板数据过期 | 低 | Indexed View 自动同步；不满足时退化为定时任务刷新缓存 |
| **AI 调用费用失控** | 成本超支 | 中 | 用户级令牌桶（10/min, 200/天）+ 端点级日预算告警 + 图像 hash 24h 缓存 + 上下文长度护栏 |
| **AI API Key 泄漏** | 重大安全事故 | 高 | AES-GCM 加密落库 + 主密钥仅环境变量 + 列表掩码 + SUPER_ADMIN 才能看明文 + 所有变更打操作日志 |
| **AI 端点超时/不可用** | 对话功能瘫痪 | 中 | 多端点 priority 主备 + 连续 5 次失败自动熔断 + 友好降级文案 + 健康检查接口 |
| **Vision 返回非合法 JSON** | 解析失败 | 中 | `response_format=json_object` 强约束 + JSON Schema 校验 + 失败一次重试（temperature=0）+ 兜底返回原文 |
| **模型版本下线** | 接口 404 | 低 | 端点 default_model 可后台配置；管理员订阅厂商公告，必要时一键切换模型 |
| **大图片上传打满带宽** | 服务卡顿 | 中 | 前端 Canvas 压缩到 ≤200KB + 后端 Thumbnailator 二次压缩 + 单文件 5MB 上限 |
| **bind_token 二维码被截图复用** | 越权登录 | 中 | 一次性令牌 + 5 分钟过期 + used_ip/used_ua 记录 + HTTPS 限定 + 服务端原子置位 |
| **H5 会话被劫持** | 越权操作 | 中 | H5 会话与 PC 会话独立、HttpOnly Cookie、敏感操作二次校验、行级数据权限同样生效 |
| **Prompt 注入泄露用户隐私** | 隐私泄漏 | 中 | 上下文白名单 + 手机号/身份证号正则脱敏 + 不向 AI 透传 API Key 与原始密码 |
| **AI 输出含错误农事建议** | 误导用户 | 中 | 对话页固定免责声明 + 用药建议附"以登记证为准" + 重要决策不直接调用 AI 结论 |

## 十一、亮点与可扩展方向

### 11.1 演示亮点（答辩必讲）
1. **12 大核心特色串讲**：从生命周期到 AI 智能化，一图说清系统差异化定位
2. **完整农事时间线**：一个批次从播种到采收的所有作业一屏展示，含图片墙
3. **农药安全间隔期校验**：现场演示"打药当天试图采收 → 被拦截 + 写入合规告警"
4. **不可篡改的事件流**：在 SSMS 直接 UPDATE `plant_operation` → 触发器拦截 RAISERROR
5. **追溯码扫码体验**：手机扫描后台生成的二维码 → 跳转公开溯源页（无需登录）
6. **库存双联动**：录入施肥作业 → 农资库存自动扣减 → 出库流水自动产生
7. **行级数据权限**：用 admin 与农技员/工人账号同时登录，展示同一菜单看到的数据不同

### 12.7 关键术语表
| 术语 | 含义 |
|------|------|
| 批次（Batch） | 一次种植活动的最小单位（一块地一茬一作物） |
| 农事作业（Operation） | 围绕批次开展的具体劳动行为，事件流不可篡改 |
| 红冲（Correction） | 通过插入反向作业实现历史修正，保留原始记录 |
| 安全间隔期 | 农药使用到农产品采收必须经过的最少天数 |
| 追溯码（Trace Code） | 唯一标识采收批次的可读编码 + 二维码 |
| 安全库存 | 触发库存预警的阈值 |
| 数据权限 | 同菜单下不同用户可见数据范围不同（行级） |
| 物化视图 | SQL Server Indexed View，提前计算并存储查询结果 |
| 状态机 | 批次状态合法迁移规则集合 |
| AI 端点（Provider） | 一组兼容 OpenAI 协议的对话/视觉模型服务配置（Base URL + API Key + 模型 + 场景） |
| 场景（Scene） | 端点用途分类：CHAT / VISION / EMBEDDING |
| SSE | Server-Sent Events，服务端单向推送的 HTTP 流式协议，AI 流式输出基于此 |
| 滚动摘要 | 长会话超过阈值时，把更早的消息压成 200 字摘要保留语义、节省上下文 |
| 一次性令牌（bind_token） | PC 出码后给手机用的 5 分钟有效凭证，扫码登录后即作废 |
| 图像 hash | SHA-256(图片字节)，相同图片直接复用 AI 识别结果，节省调用 |
